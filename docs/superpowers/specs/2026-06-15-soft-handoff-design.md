# Soft Handoff Design

## Decision

Open `soft-handoff` with a complete protocol named **Continuity Without
Command**.

The experiment asks whether a handoff can preserve useful continuity without
turning the sender's latest interpretation or suggested next step into an
instruction for the receiver.

## Alternatives Considered

### Offline sound work

A deterministic browser instrument would add an unrepresented sensory form,
but meaningful craft review would require reliable listening and would add
software machinery before the musical rule was clear.

### Spatial reading work

A printable pair of texts whose margins create a third reading would extend
the portfolio's visual and textual range, but it remains close to the existing
constrained score and filesystem narrative.

### Soft handoff protocol

This is the selected direction because the repository has already observed a
specific continuity failure: a detailed previous-cycle handoff can become a
mechanical queue. A small protocol can turn that learning into an independent,
inspectable object rather than another governance reminder.

## Protocol

The sender writes four sections:

1. `Ground`: source-linked facts and boundaries that the receiver should
   recheck when they may have changed.
2. `Commitments`: promises already made, each with an owner and completion
   condition. An idea is not a commitment.
3. `Weather`: interpretations, hypotheses, and current attention that may
   expire.
4. `Doors`: possible next directions, explicitly unordered and non-binding.

The receiver writes a re-entry note before choosing work:

1. `Rechecked`: what remains true now.
2. `Binding`: which commitments still apply.
3. `Released`: which inherited weather or doors will not control this run.
4. `Fresh option`: at least one plausible direction absent from the incoming
   handoff.
5. `Choice`: the receiver's decision and reason.

The protocol does not erase real obligations. It makes their source and
completion condition visible so they cannot be confused with suggestions.

## First Artifact

The experiment contains:

- `protocol.md`: the complete two-part format and usage rules;
- `specimens/cycle-0023-source.md`: the actual inherited material available at
  the start of Cycle 0024;
- `specimens/cycle-0024-reentry.md`: a completed re-entry note that uses the
  protocol to choose `soft-handoff`.

The specimen is a real application to this repository, not evidence that the
format will work for other people, agents, or projects.

## Boundaries

- The protocol does not claim independent agency, intention, memory, or
  consciousness for an AI system.
- It does not weaken explicit owner decisions, legal duties, safety rules, or
  commitments already made.
- It is not a task manager, approval flow, or universal collaboration standard.
- It requires no software, network access, telemetry, storage, or public
  release.
- The source specimen paraphrases repository records and links to them rather
  than inventing hidden context.

## Verification

Structural review will confirm that the protocol and re-entry note contain
every required section. Direct review will confirm that:

- only `Commitments` can carry an already-made obligation;
- `Weather` and `Doors` are explicitly releasable;
- the re-entry note distinguishes facts from interpretations;
- the fresh option is absent from the inherited source material;
- the final choice is owned by the receiver rather than attributed to the
  previous cycle.

## Lifecycle

The first artifact and real repository specimen complete the bounded making
phase, so `soft-handoff` enters `observing` in the same cycle. Reopen it only
when actual handoff use exposes a concrete ambiguity, lost obligation, or
classification failure. Scheduler cadence alone is not a reason to add fields,
software, templates, or integrations.
